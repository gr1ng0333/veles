# Ouroboros Context Builder (`context.py`)

## Назначение
Система сборки сложного текстового контекста, из которого состоит `System Prompt` агента в момент запроса к LLM. Вынесено из `agent.py` для чистоты архитектуры.

## Очередность сообщений и Prompt Caching
Ouroboros оптимизирован под Prompt Caching (особенно Claude 3.5 Sonnet). Контекст разбивается на 3 логических блока для кэширования:
1. **Static Block** (кэшируется на 1 час): `SYSTEM.md` + `BIBLE.md` + `README.md`.
2. **Semi-stable Block** (кэшируется): Identity + Scratchpad + Индекс Базы Знаний. Меняются достаточно редко, поэтому выгоднее переписывать их пореже.
3. **Dynamic Block** (НЕ кэшируется, пересчитывается каждый раз): Состояние диска (`state.json`), сводка последних метрик LLM (цена текущей задачи и т.д.), здоровье системы (`Health Invariants`), и выжимка из логов событий.

## Самодиагностика (Health Invariants)
Внутри функции `_build_health_invariants` "спрятаны" механизмы LLM-first проверок (реализация Bible Principle 0):
Код сам ничего не ломает, но он инжектирует в системный промпт страшные красные предупреждения ("CRITICAL", "WARNING"), если замечает:
- Версия в `pyproject.toml` не бьется с git tegs.
- Слишком старый файл `identity.md` (агент давно себя не рефлексировал).
- Подозрительное дублирование: одно и то же сообщение юзера попало в разные таски.

## Soft Cap и компрессия ходов
Содержит алгоритмы "обрезки", чтобы не вылететь за лимит 200,000 токенов (через `apply_message_token_soft_cap`). 
Также здесь лежат алгоритмы `compact_tool_history` — если история вызовов инструментов стала слишком длинной (много циклов-размышлений), `context.py` заменяет выхлоп старых тулзов на микро-строки из разряда `"... (truncated 5000 chars)"`, оставляя LLM только суть старых действий.
