## 1. Какие модели используются и зачем

В системе есть **4 слота** для моделей. Каждый слот отвечает за свой тип работы:

### 1.1 `OUROBOROS_MODEL` — Основная модель (мозг)

**Что делает:** Это главная рабочая лошадка. Все задачи от пользователя (написать код, ответить на вопрос, проанализировать репо) идут сюда.

**Текущее значение в .env:** `anthropic/claude-sonnet-4.6`

**Где используется:**
- В главном цикле `loop.py` — каждый вызов LLM по умолчанию использует эту модель.
- В `vision.py` — для анализа скриншотов (если не указана другая).
- В `llm.py` метод `default_model()` возвращает именно её.

### 1.2 `OUROBOROS_MODEL_LIGHT` — Лёгкая/дешёвая модель

**Что делает:** Используется для задач, где не нужен мощный ИИ, а нужно быстро и дёшево:
- Фоновое "сознание" (Background Consciousness) — размышления агента между задачами.
- Суммаризация диалогов (`summarize_dialogue` в `core.py`).
- Сжатие контекста (`compact_tool_history_llm` в `context.py`).
- Дедупликация задач (в `events.py` проверяет, не дублирует ли новая задача уже существующую).

**Текущее значение в .env:** `anthropic/claude-haiku-4.5`

**По умолчанию (если не задана):** `google/gemini-3-pro-preview`

### 1.3 `OUROBOROS_MODEL_CODE` — Модель для кода

**Что делает:** Указывает модель, используемую для задач редактирования кода. По умолчанию совпадает с основной моделью (`OUROBOROS_MODEL`).

**Текущее значение в .env:** не задана явно (по умолчанию = `anthropic/claude-sonnet-4.6`)

### 1.4 `OUROBOROS_MODEL_WORKER` — Модель для воркеров (вспомогательная)

**Текущее значение в .env:** `qwen/qwen3.5-flash-02-23`

**Примечание:** Эта переменная определена в `.env`, но в текущем коде не используется напрямую. Скорее всего, это заготовка для будущей функциональности или кастомная переменная для ручного оверрайда.

### 1.5 `OUROBOROS_MODEL_FALLBACK_LIST` — Цепочка фоллбэков

**Что делает:** Если основная модель вернула пустой ответ (бывает при перегрузке API), система автоматически пробует следующую модель из списка.

**По умолчанию:** `google/gemini-2.5-pro-preview,openai/o3,anthropic/claude-sonnet-4.6`

**Логика:** Берётся первая модель из списка, которая НЕ совпадает с текущей активной.

## 2. Переключение моделей на лету (switch_model)

Агент может сам переключить модель прямо во время выполнения задачи! Для этого существует инструмент `switch_model`.

### Как работает:
1. LLM решает, что ему нужна другая модель (например, сложный код → нужна модель помощнее).
2. LLM вызывает `switch_model(model="openai/o3", effort="high")`.
3. Значения записываются в `ToolContext` (`ctx.active_model_override`, `ctx.active_effort_override`).
4. На **следующем** раунде цикла `loop.py` подхватывает новые значения и шлёт запрос уже в другую модель.

### Уровни reasoning effort:
| Уровень | Когда использовать |
|---|---|
| `low` | Простые ответы, "да/нет", форматирование |
| `medium` | Стандартная работа (по умолчанию) |
| `high` | Сложный анализ, архитектурные решения |
| `xhigh` | Самые сложные задачи, глубокий рефакторинг |

### Доступные модели для switch_model:
Определяются в методе `available_models()` в `llm.py`. Возвращает список из:
- `OUROBOROS_MODEL` (основная)
- `OUROBOROS_MODEL_CODE` (если отличается от основной)
- `OUROBOROS_MODEL_LIGHT` (если отличается от предыдущих)

## 3. Как работают воркеры (параллельные агенты)

### 3.1 Архитектура

Система двухуровневая:

```
┌─────────────────────────────────────────────┐
│           SUPERVISOR (colab_launcher.py)     │
│  Главный процесс. Крутит бесконечный цикл:  │
│  - Читает Telegram updates                  │
│  - Раздаёт задачи воркерам                  │
│  - Следит за здоровьем воркеров              │
│  - Обрабатывает события от воркеров          │
├─────────────────────────────────────────────┤
│  Worker 0   │  Worker 1   │  Worker 2  ...  │
│  (процесс)  │  (процесс)  │  (процесс)     │
│  OuroborosAgent  OuroborosAgent  ...        │
│  ↕ Queue     │  ↕ Queue    │  ↕ Queue       │
└─────────────────────────────────────────────┘
```

### 3.2 Два режима работы

#### Мультипроцессный режим (по умолчанию)
- Supervisor запускает N процессов-воркеров (`multiprocessing`).
- Каждый воркер — отдельный процесс Python с собственным экземпляром `OuroborosAgent`.
- Задачи передаются через `multiprocessing.Queue`.
- События идут обратно через `EVENT_Q`.

#### Direct-chat режим (fallback)
- Если все воркеры крашатся 3 раза за 60 секунд ("crash storm"), мультипроцессинг отключается.
- Задачи обрабатываются прямо в главном процессе Supervisor-а в отдельном потоке (`threading`).
- Это менее надёжно, но гарантирует, что бот не замолчит.

### 3.3 Жизненный цикл задачи

1. **Пользователь пишет что-то в Telegram** → Supervisor получает update.
2. **Supervisor проверяет:** "Агент свободен?" 
   - Если ДА → запускает `handle_chat_direct()` в отдельном потоке.
   - Если НЕТ (агент занят) → вызывает `agent.inject_message()`, чтобы агент увидел дополнительное сообщение между LLM-раундами.
3. **Агент работает:** вызывает LLM, получает tool_calls, выполняет инструменты, повторяет.
4. **Агент закончил:** отправляет событие `task_done` → Supervisor пересылает ответ в Telegram.

### 3.4 Очередь задач (Queue)

Задачи имеют **приоритеты**:
| Приоритет | Тип задачи |
|---|---|
| 0 (высший) | `task`, `review` — пользовательские и код-ревью |
| 1 | `evolution` — саморазвитие |
| 2 | Всё остальное |

### 3.5 Timeouts

Каждая задача контролируется двумя таймаутами:

- **Soft Timeout** (`OUROBOROS_SOFT_TIMEOUT_SEC`, по умолчанию 600 сек) — отправляет уведомление владельцу "задача работает N секунд, продолжаю".
- **Hard Timeout** (`OUROBOROS_HARD_TIMEOUT_SEC`, по умолчанию 900 сек) — убивает воркер и пересоздаёт из чистого процесса. Задача может быть поставлена в очередь на retry (максимум 1 повтор).

### 3.6 Heartbeat

Каждый агент шлёт `task_heartbeat` события в EVENT_Q. Supervisor отслеживает `last_heartbeat_at`. Если heartbeat не приходит дольше 120 секунд, воркер считается "stale" (застрявшим).

### 3.7 Auto-resume

После рестарта системы (VPS перезагрузка, crash), Supervisor проверяет:
- Есть ли содержимое в Scratchpad?
- Были ли недавние рестарты?

Если да — автоматически инжектирует задачу *"Continue your work. Read scratchpad."*, чтобы агент продолжил с того места, где остановился.

## 4. Фоновое сознание (Background Consciousness)

Отдельный поток, который живёт параллельно с задачами:
- **Просыпается** каждые N секунд (по умолчанию 120-300 сек, решает сам LLM).
- **Засыпает** когда приходит реальная задача от пользователя (`pause()`).
- **Просыпается снова** когда задача заканчивается (`resume()`).

Использует только `OUROBOROS_MODEL_LIGHT` (дешёвую модель), чтобы не сжигать бюджет.

**Бюджет:** не более `OUROBOROS_BG_BUDGET_PCT`% от `TOTAL_BUDGET`.

**Управление:** через команду `/bg` в Telegram.

## 6. Команды Telegram-бота

Все команды начинаются с `/` и обрабатываются функцией `_handle_supervisor_command` в `colab_launcher.py`.

### 6.1 Терминальные команды (полностью обрабатываются Supervisor-ом)

| Команда | Описание |
|---|---|
| `/panic` | **АВАРИЙНАЯ ОСТАНОВКА.** Убивает все воркеры, сохраняет offset и завершает процесс. Используй только в экстренной ситуации |
| `/restart` | Мягкий перезапуск. Сохраняет состояние, убивает воркеры, перезапускает весь Python-процесс через `os.execv`. Полезно после обновления кода |

### 6.2 Двойные команды (обрабатывает Supervisor + передаётся агенту)

Эти команды сначала выполняют действие на уровне Supervisor, а потом ещё прокидываются в LLM как текстовое сообщение (чтобы агент тоже знал, что произошло):

| Команда | Описание |
|---|---|
| `/status` | Показывает полный статус: воркеры, очередь, бюджет (потрачено/осталось), breakdown по моделям, версия кода, evolution |
| `/review` | Ставит в очередь задачу code-review. Агент проведёт глубокий анализ кодовой базы |
| `/evolve [on/off/start/stop]` | Включает/выключает режим Evolution — автоматическое саморазвитие агента когда он простаивает |
| `/bg [start/stop/status]` | Управление Background Consciousness. `start` — включить, `stop` — выключить, без аргументов — показать статус |

### 6.3 Обычные сообщения

Любое текстовое сообщение, не начинающееся с `/`, отправляется напрямую агенту как задача. Агент:
1. Читает Scratchpad и Identity.
2. Собирает контекст (BIBLE.md, state.json, логи).
3. Отвечает, используя доступные инструменты.

### 6.4 Фото и изображения

Если отправить фото или документ-изображение, бот:
- Скачивает файл через Telegram API.
- Конвертирует в base64.
- Передаёт агенту как multipart content (текст + изображение).
- Агент использует Vision-модель для анализа.

**Важно:** Если агент занят задачей, фото НЕ будет обработано. Бот ответит: "📎 Photo received, but a task is in progress. Send again when I'm free."

### 6.5 Батчинг сообщений

Если ты быстро отправляешь несколько сообщений подряд (в течение 1.5 секунд), бот **склеит** их в одно и отправит агенту как единый запрос. Это предотвращает ситуацию, когда агент начинает работу по первому слову, а потом получает "отмену" вторым сообщением.

## 7. Прайсинг и учёт расходов

### Откуда берутся цены?
1. При старте системы `loop.py` пытается скачать актуальные цены с `https://openrouter.ai/api/v1/models`.
2. Если не удалось — используется зашитая таблица `_MODEL_PRICING_STATIC`.
3. Каждые 50 LLM-вызовов система проверяет "ground truth" через OpenRouter API `/auth/key`, чтобы сравнить свой подсчёт с реальным.

### Budget drift
Если разница между нашим подсчётом и OpenRouter > 50% при абсолютной разнице > $5, в промпт агента инжектируется предупреждение `CRITICAL: BUDGET DRIFT`.

## 8. Быстрая шпаргалка

```
TOTAL_BUDGET=20                     # Сколько долларов можно потратить
OUROBOROS_MODEL=anthropic/claude-sonnet-4.6    # Основная модель
OUROBOROS_MODEL_LIGHT=anthropic/claude-haiku-4.5  # Дешёвая для фона
OUROBOROS_MAX_WORKERS=1             # 1 воркер (для VPS достаточно)
OUROBOROS_MAX_ROUNDS=100            # Макс раундов на задачу
OUROBOROS_SOFT_TIMEOUT_SEC=300      # 5 мин — уведомление
OUROBOROS_HARD_TIMEOUT_SEC=900      # 15 мин — убить задачу
OUROBOROS_BG_BUDGET_PCT=10          # 10% бюджета на "мысли"
```

**Команды в Telegram:**
- Написать любое сообщение → агент обработает как задачу
- `/status` → показать всё
- `/panic` → аварийный стоп
- `/restart` → перезапуск
- `/evolve on/off` → саморазвитие
- `/bg start/stop` → фоновое сознание
- `/review` → заказать ревью кода
