# 2026-03-23 — Short-video pack pipeline: audit and design note

## Goal

Собрать минимальный контур для curated-пакетов short-video ссылок:
- **Veles** сам ищет и отбирает ролики;
- отдельный **tool** получает уже подготовленный manifest;
- tool **скачивает**, **валидирует**, **отправляет** результат владельцу в Telegram;
- контур переживает частичные фейлы и не превращается в монолитный `find+rank+download+deliver`.

## Что уже есть в кодовой базе

### 1. Доставка файлов в Telegram уже существует
Файл `ouroboros/tools/core.py` уже даёт три нужных кирпича:
- `send_document` — отправка одного файла;
- `send_documents` — пакетная последовательная отправка нескольких файлов;
- `send_local_file` — отправка уже существующего локального файла из `repo`, `drive_root` или `/tmp`.

Практический вывод: **новому downloader-tool не нужно самому кодировать Telegram transport**. Достаточно скачать файлы в допустимую локальную директорию и вызвать уже существующий путь доставки.

### 2. Архивирование артефактов уже есть
Файл `ouroboros/artifacts.py` уже умеет сохранять owner-facing артефакты в `artifacts/outbox/...` вместе с `.meta.json`.

Практический вывод: у нового контура уже есть готовый слой provenance:
- что было скачано;
- когда;
- каким task;
- что именно отправлено владельцу.

### 3. Браузерный контур уже есть, но он не должен смешиваться с downloader-layer
Файл `ouroboros/tools/browser.py` уже закрывает браузерную часть: навигацию, клики, форму, скриншоты.

Практический вывод: **поиск и курация** могут жить в браузерном режиме, но downloader-tool не должен сам ничего искать. Граница остаётся жёсткой:
- браузер = вкус / поиск / отбор;
- downloader = механика скачивания и доставки.

### 4. Регистрация нового tool уже бесплатна
`ouroboros/tools/registry.py` автодискаверит модули через `get_tools()`.

Практический вывод: для нового tool достаточно добавить отдельный модуль в `ouroboros/tools/`; ручного патча registry не требуется.

## Что сейчас отсутствует

### 1. Нет downloader backend для short-video URL
По репозиторию не найдено:
- `yt-dlp` / `yt_dlp`;
- `youtube-dl`;
- готового helper для скачивания TikTok/short-video URL.

Также в окружении сейчас:
- `ffmpeg` есть;
- `yt-dlp` бинарь отсутствует.

Практический вывод: нужно **честно добавить backend скачивания**, а не делать вид, что он уже есть.

### 2. Нет manifest-контракта для пакета short-video ссылок
Сейчас нет стандартного формата вида:
```json
[
  {
    "url": "https://www.tiktok.com/@user/video/...",
    "source": "tiktok",
    "title": "optional",
    "notes": "optional"
  }
]
```

Практический вывод: следующий шаг — сначала зафиксировать schema/API, а уже потом писать downloader.

### 3. Нет отдельного partial-success контура под batch download
Существующая отправка документов умеет передавать пачку файлов, но нет tool, который бы:
- дедуплицировал URL;
- скачивал элементы по одному;
- фиксировал `success/fail` по каждому элементу;
- продолжал работу при 1–2 неудачах;
- собирал итоговый report.

## Минимальный контракт первой итерации

### Вход
Новый tool принимает **либо**:
1. JSON-массив элементов manifest; **либо**
2. путь к manifest-файлу.

Каждый элемент manifest содержит:
- `url` — обязательно;
- `source` — опционально, но для первой версии полезно (`tiktok` как основной кейс);
- `title` — опционально;
- `notes` — опционально.

### Поведение
Первая версия tool:
- **не ищет** и **не ранжирует** контент;
- нормализует и дедуплицирует вход;
- скачивает элементы по одному через отдельный backend;
- валидирует, что файл реально получен и не пустой;
- отправляет владельцу либо документами, либо одним zip;
- возвращает честный итог: `N ok / M failed` + список ошибок.

### Не-цели первой итерации
Первая версия **не должна**:
- сама ходить в TikTok за поиском роликов;
- принимать решения о вкусе/релевантности;
- делать видеоредактирование;
- превращаться в universal media ingestion framework.

## Файлы-кандидаты для правки

### Точно
- `ouroboros/tools/` — новый модуль downloader-tool;
- `tests/` — новые contract tests под manifest parsing / dedupe / partial failures / report.

### Вероятно
- `pyproject.toml` — если backend скачивания потребует явную Python dependency;
- `README.md` — краткая фиксация новой capability после реализации.

## Итог аудита

Контур уже имеет хорошие кирпичи для:
- Telegram delivery,
- artifact provenance,
- browser-side curation,
- auto-registration tools.

**Главный пробел только один:** в системе нет честного downloader backend для short-video URL и нет manifest/API слоя над ним.

Значит, следующий шаг действительно должен быть не «сразу писать всё подряд», а **сначала зафиксировать manifest-формат и schema нового tool**, а потом уже встраивать скачивание и доставку.
